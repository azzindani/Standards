## 1. Security Boundary

Validation boundary theory · secrets · access control → [security](../security/STANDARDS.md). MCP tool arguments are untrusted input: they are generated by a language model from user text. The injection vectors specific to this surface:

| Vector | Rule |
|---|---|
| Path traversal | Every path parameter passes through one shared `resolve_path()` before use — resolve to absolute · reject anything outside the user's home tree · reject a disallowed extension. ✗ raw path strings in file opens, dataframe readers, or subprocess arguments |
| Subprocess injection | Subprocesses (FFmpeg, Tesseract, LibreOffice) are launched with an argument list · `shell=False` · an explicit `timeout` · captured output. ✗ `shell=True` · ✗ string-interpolated commands |
| Expression evaluation | User-supplied expressions (column calculations, filters, formulas) are parsed with an AST walker against an operator allowlist. ✗ `eval()` · ✗ `exec()` — no exception |
| Response leakage | ✗ connection strings · API keys · passwords · absolute system paths in any response, progress entry, or error message. Reference files by name only; redact credentials in errors |

## 2. Snapshot and Restore

**Every tool that modifies persistent data snapshots before writing.** No exception for "small changes" or "non-destructive edits". Applies to: files on disk (datasets, documents, models, configs) · database records (include a before-state in the response) · generated outputs that cost significant compute.

Snapshot contract:

| Rule | Detail |
|---|---|
| Location | `.mcp_versions/` beside the target file |
| Name | `{stem}_{UTC_timestamp}{ext}.bak` — UTC always |
| Collision guard | Windows timestamp resolution is coarser than microseconds → append an incrementing counter while the target name exists |
| Atomicity | Copy to a temp file in the same directory, then rename onto the final path. A crash mid-copy must never leave a partial `.bak` that looks valid |
| Return | The backup path, surfaced as `"backup"` in the tool response → [TOOLS.md](TOOLS.md) |
| Ordering | Snapshot only after input and op-array validation fully passes → [TOOLS.md](TOOLS.md). A snapshot taken before a rejected call is an orphaned file that never gets cleaned up |

`restore_version` is mandatory in every tier 1 server: restores a file from a snapshot by timestamp, delegating to the shared version-control module.

Irreversible operations (external side effects, unrecoverable transforms) additionally require an explicit `confirm: bool = False` parameter. `confirm=False` → return an error dict stating the operation is permanent and pointing at `dry_run=True`.

## 3. Output Path Safety

**✗ default `output_path` to the input file in any transform or write tool.** Defaulting the output to the input silently overwrites the user's original even though a snapshot was taken — the user still loses the file in place.

| Case | Destination |
|---|---|
| Caller passes `output_path` explicitly | That path — including when it points back at the source. `snapshot()` is the safety net |
| `output_path` omitted, input file exists | Derived sibling beside the input: `{stem}_{operation}{ext}` |
| `output_path` omitted, no input file (pure generation) | User's Downloads folder |

Derived suffixes name the operation: `_filtered` · `_merged` · `_imputed` · `_reshaped` · `_features` · `_cleaned` · `_customized`.

Every tool that writes a file returns both `"output_path"` (absolute) and `"output_file"` (name only), so a caller can chain without guessing the derived name.

Audit: any engine expression that falls back from an absent `output_path` to the input path is a silent data-loss defect. Grep for it on every review.

## 4. Companion State File

Complex files carry a companion JSON state file beside them: `{file}.mcp_state.json`. Contents: current version integer · source filename · last-modified UTC timestamp · schema map · known stats · patch history (version · timestamp · op count).

The companion file is advisory — a missing or corrupt companion never fails an operation. Rebuild it rather than error.

## 5. Operation Receipt Log

Every server that writes data maintains a persistent receipt log beside the modified file: `{file}.mcp_receipt.json`, an append-only array.

| Field | Content |
|---|---|
| `ts` | UTC timestamp of the operation |
| `tool` | Tool name that ran |
| `args` | Arguments that mattered to the change |
| `result` | One-line human-readable summary |
| `backup` | Snapshot path taken before the write |

`append_receipt` **never raises** — it wraps all I/O and drops the record silently on failure rather than crashing the operation it is recording. Receipt logging is observability, never a failure path.

Every tier 1 server exposes a `read_receipt` tool returning the log for a file.

## 6. Progress Channel

**stdout is the MCP protocol channel. ✗ print to stdout in any server or engine module** — a single print corrupts the stream. All diagnostic logging goes to stderr → [observability](../observability/STANDARDS.md).

All user-visible output travels in the `"progress"` array of the tool response:

| Rule | Detail |
|---|---|
| Presence | Every response carries `"progress"` — success and failure alike |
| Construction | Built with the shared `ok` / `fail` / `info` / `warn` / `undo` helpers. ✗ hand-built progress dicts |
| Entry shape | Status icon · short message · detail string |
| Paths | Filenames only — ✗ absolute paths in any progress message |
| Long operations | Emit intermediate `info` entries for anything taking more than a few seconds |

## 7. Resource Tiers and Constrained Mode

### CPU-first execution

**MCP tools run entirely on CPU.** The GPU belongs to the local LLM. VRAM figures below constrain the *model*, never tool execution.

- ✗ import or depend on CUDA · ROCm · Metal libraries in any engine file
- ✗ branch on GPU availability to decide whether a tool runs
- Domain libraries must work from their CPU builds
- A machine with no GPU at all must run every tool
- A domain that genuinely needs a GPU documents it as an explicit exception and still ships a CPU fallback path

### LLM capacity reference

Determines how many tool schemas fit and how much context remains per turn → [TOOLS.md](TOOLS.md).

| VRAM | Model | Quant | Context | Max tools |
|---|---|---|---|---|
| 4 GB | 3–4B | Q4_K_M | ~5K tokens | 5 |
| 6 GB | 7B | Q4_K_M | ~7K tokens | 6 |
| 8 GB | 9B | Q3_K_S | ~12K tokens | 8 |
| 12 GB | 9B | Q8_0 | ~20K tokens | 10 |
| 16 GB | 14B | Q4_K_M | ~24K tokens | 12 |
| 24 GB | 32B | Q4_K_M | ~32K tokens | 15 |

Design for the most constrained tier · test on the standard tier · document for the high end. Users with more hardware benefit through the absence of artificial limits.

### The constrained mode flag

Every server reads the `MCP_CONSTRAINED_MODE` environment variable **at call time, ✗ at import time**. Reading at call time means test monkeypatching works without module reloads, post-startup environment changes are honoured, and CI can force the small limits.

- `MCP_CONSTRAINED_MODE=1` → the constrained column of the return-size table applies → [TOOLS.md](TOOLS.md)
- Limits are only ever read through the shared `get_max_rows()` · `get_max_results()` · `get_max_depth()` helpers. ✗ hardcode a limit in an engine function
- The env var name is fixed. ✗ a project-specific alternative
- The installer sets it to `1` automatically on machines with ≤ 8 GB VRAM

### Resource-aware execution

Compute-heavy tools (training, video, large profiling) check available memory before starting and fail fast with an error dict stating required vs available GB and a hint naming a bounded-read tool to sample a subset instead.

## 8. Long-Running Operations

| Duration | Pattern |
|---|---|
| < ~30 s | Synchronous blocking is fine |
| > ~30 s | Async tool · async subprocess execution · explicit `timeout` on every call |

✗ mix sync and async tool definitions without verifying framework compatibility — either every tool is sync, or the server is set up async-aware throughout. An async tool returns a dict on every path, never `None` → [TOOLS.md](TOOLS.md).

Responses from long operations include `"elapsed_seconds"` and intermediate progress entries.

## 9. Live State and Reload

Servers that edit files open in another application attempt a best-effort reload notification through a shared live-edit module — platform-specific per OS.

**A failed reload never fails the write.** Reload failure → append an `info` progress entry telling the user to reopen the file, and return success.

## 10. Output Generation

HTML structure · theming · CSS architecture · chart bundling · offline-first JS strategy → [html_generation](../html_generation/STANDARDS.md). ✗ restate any of it here. The MCP-side contract only:

| Rule | Detail |
|---|---|
| Never stream bytes | ✗ return raw file bytes or an HTML body through the MCP channel — write to disk, return the path |
| Parameters | `output_path: str = ""` — empty means auto-name (§3) · `open_after: bool = True` — open in the default application after generation |
| Response | `"output_path"` (absolute) · `"output_file"` (name only) |
| Destination | Same directory as the input file; Downloads only when there is no input file. ✗ write beside the server source · ✗ write to system temp |
| Atomicity | Write to a temp file in the target directory, then rename. A crashed generation must never leave a half-written report at the final path |
| Auto-naming | `{input_stem}_{suffix}{ext}` — suffixes `_eda` · `_dashboard` · `_report` · `_chart` · `_distribution` · `_profile` |
| Opening | Best-effort, never fails the tool — an unopenable file still generated successfully |

## 11. Shared Data IO

### One reader per format

**✗ call a dataframe library's CSV reader directly in server or engine code.** Every CSV read routes through the single shared reader, which owns:

1. Encoding fallback chain — utf-8 → utf-8-sig (BOM) → cp1252 (Windows/Excel) → latin-1
2. Bad-line tolerance — on a tokenization error (unescaped separators producing mismatched field counts), retry skipping malformed rows
3. Column-name normalization — strip leading/trailing whitespace from headers
4. Consistent low-memory handling

A CSV from Excel carries a BOM; one from a legacy system is cp1252; one with a free-text field has an unescaped comma. Each hard-fails a bare reader call. Scattered direct calls guarantee the failure surfaces in production.

### Coercion rules

| Operation | Rule |
|---|---|
| Numeric aggregation (`mean` · `sum` · `min` · `max` · `std`) | Coerce the column to numeric with errors → null **before** aggregating. CSV numeric columns frequently arrive as strings and crash the aggregation. Report the count of coerced-to-null values as a `warn` progress entry |
| Date parsing | Always parse with mixed-format mode · day-first disabled · unparseable values coerced to null. ✗ a single fixed format string · ✗ no format at all — the fallback parser degrades to slow row-by-row parsing and fails on any variant row |

### Atomic writes

Every output file — snapshot, report, transformed dataset — is written to a temp file in the destination directory and renamed onto the target. ✗ write directly to the target path.

## 12. Cross-Platform Runtime

| Concern | Rule |
|---|---|
| Paths | Path objects and the join operator everywhere. ✗ string concatenation to build a path |
| Line endings | Repo-level attributes force LF, with CRLF for `.bat` / `.cmd` |
| Install scripts | `install.sh` targets POSIX `sh`, ✗ bash-only syntax |
| stdout | Reserved for the MCP protocol (§6) |
| macOS native libs | OpenMP-linked libraries (XGBoost, LightGBM) require `libomp`; without it the import fails outright → [DELIVERY.md](DELIVERY.md) |
| Windows long paths | Paths beyond ~200 characters need the extended-length prefix |
| Windows Defender | The real-time scanner delays every write over ~1 MB by seconds to minutes |

Defender mitigations, all three mandatory:

- HTML output stays small — no inline JS bundle → [html_generation](../html_generation/STANDARDS.md)
- Snapshots and outputs are written temp-then-rename — the scanner sees one final file, not a growing one
- Heavy libraries are imported at module level so the compiled-module scan happens at startup, before the user is waiting → [STANDARDS.md](STANDARDS.md)

## 13. Transport Modes

Every server supports two transports from one codebase.

| Mode | Use | Selection |
|---|---|---|
| stdio | Default — local AI clients | No flag |
| HTTP | Remote or multi-user access | `--transport http --port {n}` |

HTTP rules: bind to loopback by default · auth token generated by the installer (≥ 32 random hex characters) · token stored in config. ✗ hardcode a token · ✗ ship a default token.

## 14. Multi-Client Compatibility

Server code is identical across AI clients. The only difference is the location and shape of the client config file → [DELIVERY.md](DELIVERY.md). ✗ branch server logic on which client is connected.

## 15. Checklist

- [ ] Every path parameter passes through `resolve_path()` before use
- [ ] Every subprocess uses an argument list, `shell=False`, and an explicit timeout
- [ ] No `eval()` or `exec()` on user-supplied expressions anywhere
- [ ] No response, progress entry, or error leaks credentials or absolute paths
- [ ] Every write snapshots first, and only after validation passes
- [ ] Snapshots are atomic (temp + rename) and collision-guarded
- [ ] `restore_version` exists in every tier 1 server
- [ ] Irreversible operations require `confirm=True`
- [ ] No transform tool defaults `output_path` to the input file
- [ ] Every file-writing tool returns `"output_path"` and `"output_file"`
- [ ] `append_receipt` never raises; a receipt failure never fails the operation
- [ ] Every tier 1 server exposes `read_receipt`
- [ ] Zero `print()` calls in any server or engine module; logs go to stderr
- [ ] Every response carries a `"progress"` array built with the shared helpers
- [ ] No engine file imports CUDA, ROCm, or Metal; no tool branches on GPU availability
- [ ] `MCP_CONSTRAINED_MODE` is read at call time, never at import time
- [ ] Every limit comes from `get_max_*()`; no hardcoded size constants
- [ ] Compute-heavy tools check available memory and fail fast with a hint
- [ ] Operations over ~30 s are async with an explicit timeout
- [ ] A failed live-reload never fails the write
- [ ] Report tools return a path; no HTML body or raw bytes crosses the MCP channel
- [ ] Every CSV read routes through the shared reader
- [ ] Numeric aggregations coerce the column first; date parsing uses mixed format
- [ ] Every output file is written temp-then-rename
- [ ] Paths are built with path objects, never string concatenation
- [ ] Both stdio and HTTP transports work; the HTTP token is installer-generated
